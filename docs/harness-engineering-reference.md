# Harness Engineering Reference (랄프톤 우승팀 분석)

Source: 브라이언(장동욱) - "한국 최초 랄프톤 후기 : 하네스 엔지니어링 시대" (2026-03-02)

## 우승팀 핵심 전략

### 1. 코드 비율: 10만줄 중 7만줄이 테스트 (70%)
- 에이전트를 수시간 동안 믿고 돌려야 하니 단단한 테스트 코드 필수
- 밸리데이터, 코디네이터, 패커 에이전트가 모호함을 없애고 거짓 성공을 걸러냄

### 2. 소크라틱 리즈닝 133회 → Ambiguity 0.05
- PRD/스펙의 모호함을 없애기 위해 에이전트가 스스로 질문하는 문답법 133회 수행
- Ambiguity = 1 - SUM(clarity_i * weight_i)
- 0.05까지 낮춘 후 랄프모드 시작
- 이것이 키보드를 한 번도 만지지 않을 수 있었던 핵심 비결

### 3. 스펙 확장 절대 금지
- 코딩 중 스펙을 급하게 확장한 팀은 배포가 멈추고 이상한 루프에 빠짐
- 모호성 증가 → 자가 검증 불가 → 꼬임

### 4. Self-Healing (구현 우회)
- 5번 이상 테스트 실패 시 요구사항은 타협 없이 구현 방식만 완전 교체
- 라이브러리 교체, 아키텍처 우회 등

### 5. I Ship Code I Don't Read
- 사람이 읽기 좋은 코드가 아니라 에이전트가 이해하고 고치기 좋은 구조
- 주석/포맷팅보다 기계 친화적 구조화

## 실패 패턴
- 너무 빨리 끝날 것 같아 스펙 급확장 → 모호성 폭발 → 루프 멈춤
- 워크트랙 3개 병렬 → 배포 멈춤 → 수동 개입 필요

## 하네스 엔지니어링 실행 프롬프트

### Phase 1: 소크라틱 리즈닝 & 백로그 생성
- 코딩 전 스스로 30번 이상 문답
- Ambiguity 0.05 이하까지
- 엣지 케이스, API 리밋, DB 트랜잭션 충돌, 데이터 로깅 유실 등 모든 예외 방어
- 산출물: 고정된 50개 초세밀 마일스톤, 70% 테스트 명세서, 기계 친화적 PRD

### Phase 2: 무한 하네스 랄프 루프
1. 스펙 확장 절대 금지
2. 70% Test-Driven (테스트 먼저)
3. Validation (거짓 성공 방지 - 실제 실행 결과만 인정)
4. Self-Healing (5회 실패 시 구현 방식 교체, 스펙은 불변)
5. I ship code I don't read (기계 친화적 구조)
6. 마일스톤 완료 → git commit + push → 컨텍스트 비우기 → 다음 마일스톤
